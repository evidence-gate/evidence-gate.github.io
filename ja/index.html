AIが不正できない品質ゲート

AIエージェントがコード生成だけでなく、テスト作成まで担うようになった今、見えている閾値は「品質基準」ではなく「最適化の目標」になりがちです。Evidence Gateは、Blind Gatesで判定基準をAIから隠したまま評価し、フェイルクローズドですべてのゲートを守ります。SBOM、来歴、テストカバレッジなど32種のゲートタイプに対応。

仕組み

3ステップで、プルリクエストに品質基準を適用

1

定義

GitHub ActionsのワークフローにEvidence Gateを追加し、ゲートタイプ、対象ファイル、評価条件を指定します。

2

評価

Evidence Gateがエビデンスファイルを自動検証します。ファイルの有無、スキーマ適合性、閾値、整合性などを確認します。

3

強制

基準を満たさない場合はパイプラインを停止します。結果はPRサマリーやワークフローアノテーションで確認できます。

Blind Gates

なぜ、AIエージェントに判定基準を見せないのか

LLMがコードだけでなくテストまで生成する状況では、見えている閾値は「品質基準」ではなく「最適化の目標」になりがちです。

問題: 従来のCIゲートでは、たとえばテストカバレッジの閾値がワークフローYAMLに書かれています。AIコーディングエージェントはその数値を読み取り、ぎりぎり通過するためだけの空疎なテストを作ることができます。数値は満たしていても、品質を保証しているとは言えません。

解決策: Blind Gatesでは、評価基準をパイプラインやリポジトリ、AIエージェントに公開しません。パイプラインはエビデンスファイルを提出し、Evidence Gate APIがチームごとの非公開基準で評価します。返される結果はPASSかFAILだけです。基準そのものは公開されないため、AIが閾値に合わせて振る舞うことはできません。

AIガバナンスを前提にした設計

Evidence Gateは、透明性、安全性、説明責任といったAIガバナンスの基本原則に沿って設計されています。

フェイルクローズド

すべてのゲートはデフォルトでFAILです。明示的に検証されたエビデンスだけがPASSになります。

透明性と信頼レベル

エビデンスの信頼性をL1〜L4のレベルで整理できます。SHA-256によるエビデンスチェーンで、判断に使ったデータの整合性も確認できます。

セキュリティと説明責任

AWS KMSによる暗号化、HMAC署名カーソル、成熟度ベースの品質状態モデルにより、各ステップを監査可能な形で管理できます。

5分で導入

1 Marketplaceから追加

2 ワークフローに追加

GitHub ActionsにEvidence Gateステップを設定します。

name: Quality Gate
on: [pull_request]

permissions:
  contents: read
  checks: write

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Your build & test steps here...

      - name: Evidence Gate
        uses: evidence-gate/evidence-gate-action@v1
        with:
          # Or use .evidencegate.yml config file for zero required inputs
          gate_type: "test_coverage"
          phase_id: "testing"
          evidence_files: "coverage.json"

NVIDIA NemoClaw + Evidence Gate

NemoClawサンドボックスとの統合

Evidence Gateの品質ゲートは、CIパイプラインだけでなく、NemoClawサンドボックスのブループリント、ポリシー、推論設定まで検証できます。ここからは、NemoClawとの統合について説明します。

Evidence Gateが守るもの

CI上のチェックだけでは不十分です。AIエージェントは、実行時にもリスクを持ち込みます。Evidence Gateは、AIエージェントが動作するNemoClawインフラまで含めて検証します。

パイプラインを通過した後、AIエージェントはNemoClawサンドボックス上で実行されます。このサンドボックスに設定ミスがあると、隔離が不十分になったり、リソース消費が制御不能になったりするおそれがあります。そのためEvidence Gateは、ブループリント、ポリシー、推論設定をデプロイ前に検証します。

セキュリティ保証

Evidence Gateは、サンドボックスが4つの必須分離レイヤーを満たしているかを検証します。どれか1つでも不備があれば、パイプラインを停止します。

Landlock LSM

カーネルレベルでファイルシステムアクセスを制御します。許可されたパス以外にはアクセスできません。

seccomp Filtering

危険なシステムコールを制限し、不要なカーネル操作をブロックします。

Network Namespace Isolation

各サンドボックスは独立したネットワーク名前空間で動作し、通信はデフォルト拒否です。許可されたエンドポイントにのみ接続できます。

Inference Control

LLM推論はOpenShellゲートウェイ経由で行われます。モデルの許可リスト、レート制限、コスト上限を適用できます。

ファイルシステム
  • /sandbox — 読み書き可能な作業ディレクトリ
  • /tmp — 一時ファイル用
  • その他すべてのパス — 読み取り専用、またはアクセス不可
ネットワーク
  • 通信はデフォルト拒否
  • 許可先はblueprint.yamlに明示
  • 未承認リクエストは遮断し、オペレーター確認に回す
  • 推論トラフィックはゲートウェイ経由のみ

Evidence Gateのblueprintゲートとpolicyゲートは、これら4つの分離レイヤーが本番投入前に正しく構成されていることを確認します。

NemoClaw統合クイックスタート

NemoClaw設定の検証、ランタイム予算の強制、すべてをCIでゲート — 1つのワークフロー

name: NemoClaw Governance
on: [pull_request]

permissions:
  contents: read
  checks: write

jobs:
  validate-blueprint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Validate NemoClaw blueprint.yaml
      - name: Blueprint Gate
        uses: evidence-gate/evidence-gate-action@v1
        with:
          gate_type: "nemoclaw_blueprint"
          evidence_files: "blueprint.yaml"

  validate-policy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Validate sandbox policy constraints
      - name: Policy Gate
        uses: evidence-gate/evidence-gate-action@v1
        with:
          gate_type: "nemoclaw_policy"
          evidence_files: "policy.yaml"

  enforce-budget:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Validate agentgov budget configuration
      - name: Budget Gate
        uses: evidence-gate/evidence-gate-action@v1
        with:
          gate_type: "custom"
          phase_id: "budget"
          evidence_files: "agentgov.config.json"

3つの並列ジョブ — ブループリント構造、サンドボックスポリシー、ランタイム予算 — すべてマージ前に検証。各ゲートはフェイルクローズド: いずれかの設定が無効であればPRがブロックされます。
